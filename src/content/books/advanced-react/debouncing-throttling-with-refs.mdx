在之前的章节中，我们详细讨论了 Ref 是什么，如何使用它，以及如何不使用它。然而，还有一个非常重要且相当常见的 Refs 用例我们尚未覆盖。那就是在处理 setInterval 或 debounce 等函数时存储各种定时器和超时 ID。这是处理各种表单元素时非常常见的场景。例如，我们通常希望对输入框的 onChange 回调进行去抖/节流，以便表单不会在每次键入时重新渲染。

在 React 中正确实现去抖/节流实际上是相当具有挑战性的。你不能仅仅将 onChange 回调用 lodash 库中的 debounce 包裹起来，然后期待一切正常。因此，在本章中，让我们尝试实现一个好的 useDebounce 钩子，看看在这个过程中会遇到什么挑战和常见的陷阱。

我将使用 Lodash 库中的 debounce 和 throttle 函数，并专注于 React 特定的内容。

在此过程中，我们将学习：

- 去抖和节流是什么，以及它们之间的区别（快速知识刷新）。

- 为什么我们不能直接在事件处理程序上使用 debounce。

- 如何使用 useMemo 或 useCallback 来实现这一点，以及它们的缺点。

- 如何使用 Refs 进行去抖，以及 Refs 与使用 useMemo 和 useCallback 之间的区别。

- 如何使用逃避闭包陷阱的技巧来实现去抖。

### 去抖和节流是什么？

让我们简要回顾一下“去抖”和“节流”是什么，以防你还没有机会使用它们。“去抖”和“节流”是技巧，它们允许我们在一定时间段内跳过函数执行，如果该函数被调用的次数过多。

想象一下，我们正在实现一个简单的异步搜索功能：一个输入字段，用户可以在其中键入内容，同时该文本被发送到后端，后端返回相关的搜索结果。我们当然可以“天真地”实现它，只需一个输入字段和 onChange 回调：

```jsx
const Input = () => {
  const onChange = (e) => {
    // send data from input field to the backend here
    // will be triggered on every keystroke
  };
  return <input onChange={onChange} />;
};
```

但是，一个熟练的打字员可以以每分钟 70 个单词的速度输入，这大约是每秒 6 次按键。在这种实现中，将会导致每秒 6 次 onChange 事件，即每秒 6 次请求服务器！并不是每个后端都能处理这种情况，也没有必要。

我们可以等一等，直到用户停止输入，然后一次性发送整个值，而不是在每次按键时发送请求。这就是去抖的作用。如果我对 onChange 函数应用去抖，它会检测我每次尝试调用它，如果等待间隔尚未过去，它会丢弃之前的调用并重新启动“等待”计时器。

```jsx
interval hasn't passed yet, it will drop the previous call and restart the
"waiting" clock.
const Input = () => {
 const onChange = (e) => {
 // send data from input field to the backend here
 // will be triggered 500 ms after the user stopped typing
 };
 const debouncedOnChange = debounce(onChange, 500);
 return <input onChange={debouncedOnChange} />;
};
```

以前，如果我在搜索字段中键入“React”，请求会在每次按键时立即发送，值为“R”、“Re”、“Rea”、“Reac”、“React”。而现在，在去抖之后，它会在我停止输入“React”后等待 500 毫秒，然后仅发送一次带有值“React”的请求。

在底层，去抖只是一个接受一个函数并返回另一个函数的函数，内部有一个跟踪器，用于检测传递的函数是否在提供的间隔内被调用。如果太快——则跳过执行并重新启动计时器。如果间隔已经过去，则调用传递的函数。基本上，它类似于这样：

```jsx
const debounce = (callback, wait) => {
  // initialize the timer
  let timer;
  // lots of code here involving the actual implementation of timer
  // to track the time passed since the last callback call
  const debouncedFunc = () => {
    // checking whether the waiting time has passed
    if (shouldCallCallback(Date.now())) {
      callback();
    } else {
      // if time hasn't passed yet, restart the timer
      timer = startTimer(callback);
    }
  };
  return debouncedFunc;
};
```

实际实现当然会复杂一些。你可以查看 lodash 的去抖代码，以了解其中的细节。

节流与此非常相似，保持内部跟踪器和返回函数的函数的理念是相同的。不同之处在于，节流保证在每个等待间隔内定期调用回调函数，而去抖则会不断重置计时器并等待结束。

如果我们使用的是一个带有自动保存功能的编辑字段，而不是异步搜索示例，差异将会非常明显：如果用户在字段中键入内容，我们希望发送请求到后端，以“动态”保存他们所输入的内容，而无需用户明确按下“保存”按钮。如果用户在这样的字段中非常快速地写诗，去抖的 onChange 回调只会被触发一次。如果在输入时出现问题，整首诗将会丢失。而节流的回调将会定期被触发，诗歌将会定期保存，如果发生灾难，只有最后几毫秒的内容会丢失。这是一种更安全的方法。

### 在 React 中的去抖回调：处理重新渲染。

现在，对于去抖和节流的概念、需要它们的原因以及如何实现它们的理解更加清晰了，是时候深入研究它们在 React 中的使用。希望你不要认为“这有什么难的，这只是一个函数”，对吧？生活中总会有一些意外出现。

首先，让我们仔细看看具有去抖 onChange 回调的 Input 实现（从现在开始，我将所有示例中只使用去抖；所有描述的概念也同样适用于节流）。

```jsx
const Input = () => {
  const onChange = (e) => {
    // send data from input to the backend here
  };
  const debouncedOnChange = debounce(onChange, 500);
  return <input onChange={debouncedOnChange} />;
};
```

虽然这个例子工作得非常完美，看起来就像正常的 React 代码，没有任何问题，但不幸的是，这与现实生活无关。在现实中，你可能希望对输入的值做一些其他事情，而不仅仅是将其发送到后端。也许这个输入将是一个大表单的一部分，或者你想在其中引入一个“清除”按钮，或者也可能输入标签实际上是某个外部库的组件，强制要求提供值字段。

我想说的是，在某个时刻，你会希望将那个值保存到状态中，无论是在 Input 组件内部还是将其传递给父级/外部状态管理来处理。为了简化，我们在 Input 组件中进行处理。

```jsx
const Input = () => {
 // adding state for the value
 const [value, setValue] = useState();
 const onChange = (e) => {};
 const debouncedOnChange = debounce(onChange, 500);
 // turning input into controlled component by passing value from
state there
 return (
 <input onChange={debouncedOnChange} value={value} />
 );
};
```

我通过 useState 钩子添加了一个状态值，并将该值传递给输入字段。剩下的一件事是让输入在输入时更新该状态。否则，输入将无法工作。通常，在没有去抖的情况下，这将在 onChange 回调中完成：

```jsx
const Input = () => {
  const [value, setValue] = useState();
  const onChange = (e) => {
    // set state value from onChange event
    setValue(e.target.value);
  };
  return <input onChange={onChange} value={value} />;
};
```

在去抖的 onChange 中，我不能这样做：它的调用本质上是延迟的，因此状态中的值不会及时更新，输入根本无法工作。

```jsx
const Input = () => {
  const [value, setValue] = useState();
  const onChange = (e) => {
    // just won't work, this callback is debounced
    setValue(e.target.value);
  };
  const debouncedOnChange = debounce(onChange, 500);
  return <input onChange={debouncedOnChange} value={value} />;
};
```

我必须在输入调用其自己的 onChange 时立即调用 setValue。这意味着我不能再在整体上对我们的 onChange 函数进行去抖，只能对我确实需要减缓的部分进行去抖：发送请求到后端。

听起来很合理,对吧?

```jsx
const Input = () => {
 const [value, setValue] = useState();
 const sendRequest = (value) => {
 // send value to the backend
 };
 // now send request is debounced
 const debouncedSendRequest = debounce(sendRequest, 500);
 // onChange is not debounced anymore, it just calls debounced
function
 const onChange = (e) => {
 const value = e.target.value;
 // state is updated on every value change, so input will work
```

只是……它也不工作！现在请求根本没有去抖，只是稍微延迟了一下。如果我在这个字段中输入“React”，我仍然会发送所有的“R”、“Re”、“Rea”、“Reac”、“React”请求，而不是像正确的去抖函数那样只发送一次“React”，只延迟半秒。

互动示例和完整代码 https://advanced-react.com/examples/11/02

答案当然是重新渲染（通常在 React 中是这样）。正如我们在第一章中所了解的，状态变化将导致组件重新渲染。通过引入状态来管理值后，我们现在在每次按键时重新渲染整个 Input 组件。因此，在每次按键时，我们现在调用的是实际的去抖函数，而不仅仅是去抖的回调。正如我们刚才讨论的，去抖函数在被调用时：

创建一个新计时器

创建并返回一个函数，在计时器结束时调用传递的回调

因此，当我们在每次重新渲染时调用 debounce(sendRequest, 500) 时，我们重新创建了一切：新的调用、新的计时器、新的返回函数和参数中的回调。但旧的函数从未被清理，所以它仍然在内存中等待其计时器的结束。

当它的计时器结束时，它会触发回调函数，然后就消亡，最终被垃圾回收器清理。

我们最终得到的只是一个简单的延迟函数，而不是一个真正的去抖。现在的修复方案应该显而易见：我们应该只调用一次 debounce(sendRequest, 500)，以保留内部计时器和返回的函数。

最简单的方法就是将其移到 Input 组件外部：

```jsx
const sendRequest = (value) => {
  // send value to the backend
};
const debouncedSendRequest = debounce(sendRequest, 500);
const Input = () => {
  const [value, setValue] = useState();
  const onChange = (e) => {
    const value = e.target.value;
    setValue(value);
    // debouncedSendRequest is created once, so state caused rerenders won't affect it anymore
    debouncedSendRequest(value);
  };
  return <input onChange={onChange} value={value} />;
};
```

然而，如果这些函数依赖于组件生命周期内发生的某些事情，即状态或属性，这将无效。不过没关系，我们可以使用记忆化钩子来实现完全相同的结果：

```jsx
const Input = () => {
     const [value, setValue] = useState('initial');
 // memoize the callback with useCallback
 // we need it since it's a dependency in useMemo below
 const sendRequest = useCallback((value: string) => {
 console.log('Changed value:', value);
 }, []);
 // memoize the debounce call with useMemo
 const debouncedSendRequest = useMemo(() => {
 return debounce(sendRequest, 1000);
 }, [sendRequest]);
 const onChange = (e) => {
 const value = e.target.value;
 setValue(value);
 debouncedSendRequest(value);
 };
 return <input onChange={onChange} value={value} />;
};
```

### 在 React 中的去抖回调：处理内部状态

现在来解决这个复杂的拼图的最后一块。让我们看看这段代码：

```jsx
const sendRequest = useCallback((value: string) => {
 console.log('Changed value:', value);
}, []);
```

一个普通的记忆化函数，接受值作为参数，然后对其进行处理。该值直接通过去抖函数从输入中传递。我们在 onChange 回调中调用去抖函数时传递它：

```jsx
const onChange = (e) => {
  const value = e.target.value;
  setValue(value);
  // value is coming from input change event directly
  debouncedSendRequest(value);
};
```

但我们在状态中也有这个值。我能不能直接从那里使用它？也许我有一系列这样的回调，反复传递这个值真的很麻烦。也许我想访问另一个状态变量。将其通过回调传递似乎没有意义。或者我只是讨厌回调和参数，想要仅仅使用状态。应该足够简单，对吧？

当然，再一次，事情并没有看起来那么简单。如果我只是去掉参数，从状态中使用这个值，我必须将其添加到 useCallback 钩子的依赖项中：

```jsx
const Input = () => {
  const [value, setValue] = useState("initial");
  const sendRequest = useCallback(() => {
    // value is now coming from state
    console.log("Changed value:", value);
    // adding it to dependencies
  }, [value]);
};
```

因此，sendRequest 函数将随着每次值的变化而变化。这就是记忆化的工作原理。该值在重新渲染过程中保持不变，直到依赖项发生变化。这意味着我们的记忆化去抖调用现在也会不断变化：它将 sendRequest 作为依赖项，而 sendRequest 每次状态更新时都会变化。

```jsx
// this will now change on every state update
// because sendRequest has dependency on state
const debouncedSendRequest = useMemo(() => {
  return debounce(sendRequest, 1000);
}, [sendRequest]);
```

我们又回到了引入状态到 Input 组件时的情况：去抖变成了简单的延迟。

这里有什么办法可以解决吗？当然！这是使用 Refs 的完美用例。如果你搜索有关去抖和 React 的文章，一半会提到 useRef 作为避免在每次重新渲染时重新创建去抖函数的方法。

通常，模式是这样的：

```jsx
const Input = () => {
  // creating ref and initializing it with the debounced backend
  call;
  const ref = useRef(
    debounce(() => {
      // this is our old "debouncedSendRequest" function
    }, 500),
  );
  const onChange = (e) => {
    const value = e.target.value;
    // calling the debounced function
    ref.current();
  };
};
```

这实际上可能是一个很好的替代方案，基于使用 useMemo 和 useCallback 的解决方案。我不知道你怎么想，但这些钩子的链条有时让我感到头疼。基于 ref 的解决方案似乎简单得多。

不幸的是，它只适用于之前的用例：当我们在回调中没有状态时。记住前一章和闭包的问题吗？Ref 的初始值是缓存的，永远不会更新。它在组件挂载和 ref 初始化时是“冻结”的。

正如我们已经知道的，当在 Refs 中使用函数时，我们需要在 useEffect 中更新它们。否则，闭包会变得陈旧。

```jsx
const Input = () => {
  const [value, setValue] = useState();
  // creating ref and initializing it with the debounced backend
  call;
  const ref = useRef(
    debounce(() => {
      // send request to the backend here
    }, 500),
  );
  useEffect(() => {
    // updating ref when state changes
    ref.current = debounce(() => {
      // send request to the backend here
    }, 500);
  }, [value]);
  const onChange = (e) => {
    const value = e.target.value;
    // calling the debounced function
    ref.current();
  };
};
```

但不幸的是，这与 useCallback 的依赖解决方案没有什么不同：每次都会重新创建去抖动的函数，定时器每次也会重新创建，去抖动不过是一个重命名的延迟。

互动示例和完整代码 https://advanced-react.com/examples/11/05

解决这个问题的一种方法是在 useEffect 中使用清理函数（我们将在第 15 章讨论这些内容：数据获取和竞争条件），并在重新分配之前重置去抖动的回调。类似这样的：

```jsx
useEffect(() => {
  // updating ref when state changes
  ref.current = debounce(() => {}, 500);
  // cancel the debounce callback before
  return () => ref.current.cancel();
}, [value]);
```

在这种情况下，每次更新时，我们都摆脱了“旧”的去抖动闭包，重新开始一个新的。这是去抖动的好解决方案。但不幸的是，它对节流不起作用。如果我不断取消它，它将永远没有机会在它应该被触发的间隔后被触发，因为节流就是这样。

我想要一个更通用的解决方案。另一个很好的用例是解决闭包陷阱的问题，我们在前一章中详细探讨过！我们需要做的就是将 sendRequest 分配给 Ref，在 useEffect 中更新该 Ref，以便访问最新的闭包，然后在我们的闭包中触发 ref.current。记住：refs 是可变的，闭包不会进行深度克隆。只有对该可变对象的引用是“冻结”的，我们仍然可以在每次更新时修改它所指向的对象。

思考闭包让我头疼，但这实际上有效，并且在代码中更容易跟随这个思路。

```jsx
const Input = () => {
 const [value, setValue] = useState();
 const sendRequest = () => {
 // send request to the backend here
 // value is coming from state
 console.log(value);
 };
 // creating ref and initializing it with the sendRequest function
 const ref = useRef(sendRequest);
 useEffect(() => {
 // updating ref when state changes
      // now, ref.current will have the latest sendRequest with
access to the latest state
 ref.current = sendRequest;
 }, [value]);
 // creating debounced callback only once - on mount
 const debouncedCallback = useMemo(() => {
 // func will be created only once - on mount
 const func = () => {
 // ref is mutable! ref.current is a reference to the latest
sendRequest
 ref.current?.();
 };
 // debounce the func that was created once, but has access to
the latest sendRequest
 return debounce(func, 1000);
 // no dependencies! never gets updated
 }, []);
 const onChange = (e) => {
 const value = e.target.value;
 // calling the debounced function
 debouncedCallback();
 };
};
```

现在，我们要做的就是将这种令人心烦的闭包逻辑提取到一个小钩子中，放在一个单独的文件中，然后假装没有注意到它。

```jsx
const useDebounce = (callback) => {
  const ref = useRef();
  useEffect(() => {
    ref.current = callback;
  }, [callback]);
  const debouncedCallback = useMemo(() => {
    const func = () => {
      ref.current?.();
    };
    return debounce(func, 1000);
  }, []);
  return debouncedCallback;
};
```

然后我们的生产代码可以直接使用它，而不用担心眼花缭乱的 useMemo 和 useCallback 链，也不必担心依赖关系，并且可以在里面访问最新的状态和道具！

```jsx
const Input = () => {
  const [value, setValue] = useState();
  const debouncedRequest = useDebounce(() => {
    // send request to the backend
    // access to the latest state here
    console.log(value);
  });
  const onChange = (e) => {
    const value = e.target.value;
    setValue(value);
    debouncedRequest();
  };
  return <input onChange={onChange} value={value} />;
};
```

### 关键要点

这很有趣，不是吗？JavaScript 的闭包可能是互联网上最受欢迎的特性。下一章，我们将尝试从处理它们中恢复过来，并玩弄一些 UI 改进。更具体地说，我们将学习如何消除定位元素的“闪烁”效果。但在此之前，让我们快速回顾一下本章内容：

我们在希望跳过一些执行过于频繁的函数时使用去抖动和节流。
为了使这些函数正常工作，它们应该在组件的生命周期中只调用一次，通常是在挂载时。
如果我们直接在组件的渲染函数中调用它们，内部的定时器将会在每次重新渲染时重新创建，函数将无法按预期工作。
为了解决这个问题，我们可以使用 useMemo 或通过使用 Refs 进行记忆。
如果我们简单地记忆它们或“天真”使用 Refs，我们将无法访问组件的最新数据，例如状态或道具。这是因为在初始化 Ref 时会创建一个闭包，这会冻结创建时的值。
为了逃避闭包陷阱，我们可以利用 Ref 对象的可变性质，通过在 useEffect 中不断更新 ref.current 内的“封闭”函数来获取最新数据。
